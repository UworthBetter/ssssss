# QKYD platform upgrade decision log

Record decisions here so the project does not lose architectural memory.

| Date | ID | Decision | Rationale | Impacted areas | Follow-up |
| --- | --- | --- | --- | --- | --- |
| 2026-03-22 | D-001 | Treat the upgrade as a staged platform program, not a page-by-page redesign | The main risk is losing platform intent and falling back into isolated page work | product, IA, frontend, AI | Keep all work mapped to roadmap phases |
| 2026-03-22 | D-002 | Use persistent control files in `upgrade/` to track progress and handoff | Conversation history alone is too fragile for a long-running upgrade | planning, handoff, execution | Update `upgrade/status.md` at the end of each session |
| 2026-03-22 | D-003 | Create a local reusable skill to enforce the upgrade workflow | The project needs a repeatable operating method, not only static planning docs | workflow, handoff, execution quality | Use the skill for each future upgrade session |
| 2026-03-22 | D-004 | Use five business centers as the first platform shell and remap current pages into grouped canonical routes | This is the smallest code slice that establishes platform structure without forcing a full page rewrite | router, layout, navigation, IA | Build the shared platform page shell in P1-002 |
| 2026-03-22 | D-005 | Use `ExceptionAlertView` as the first adopter of the shared platform shell and mount platform-level placeholder tools through shell slots | The event center has the strongest operational priority and the lowest migration risk among the current pages | event center, shared components, P1 shell | Extend the same shell pattern to other business pages after validating the event page |
| 2026-03-22 | D-006 | Model event-center workflow state locally in the frontend for the first `P2-001` slice, instead of waiting for backend event-entity support | This keeps the event-center upgrade moving while preserving current exception APIs | event center, workflow shell, sequencing | Replace derived workflow state with real event entity fields in a later backend-linked slice |
| 2026-03-22 | D-007 | Use the same shared-shell strategy on the subject page and introduce a right-side 360 preview before building a full detail route | This gives the product a real object-center structure with low migration risk and keeps current CRUD stable | subject center, shared shell, P3 sequencing | Expand from preview to full subject detail and real linked data later |
| 2026-03-22 | D-008 | Use the same shared-shell strategy on the device page and introduce an operations preview panel before building a dedicated device details system | This establishes an operations-oriented device center quickly while preserving current device CRUD behavior | device center, shared shell, P4 sequencing | Replace placeholder health and rule data with real backend telemetry later |
| 2026-03-22 | D-009 | Reshape the AI page into a structured decision workbench before wiring real cross-center handoff behavior | The platform needs structured evidence, entity references, and action hooks first; real handoff can then be hardened in a focused P6 slice | AI center, platform shell, cross-center linkage | Replace placeholder entity and action toasts with shared route and service behavior in P6 |
| 2026-03-22 | D-010 | Rebuild the AI, subject, device, and event center views into clean UTF-8 Vue SFCs while landing `P6-001` | Historical encoding noise had started corrupting template parsing, so route handoff hardening had to include source stabilization to keep the upgrade buildable | AI center, subject center, device center, event center, build stability | Continue the next P6 slice by extracting shared route handoff and platform service helpers |
| 2026-03-22 | D-011 | Treat the current “data chain interrupted” symptom as an auth-aware partial-failure problem instead of a pure transport outage | Live endpoint checks show the production backend is online, while unauthenticated calls return `401`; the frontend therefore needs clearer session handling and should not blank the full dashboard when one request fails | request handling, dashboard polling, operator UX | Continue P6-002 by extracting these resilience behaviors into shared platform services |
| 2026-03-22 | D-012 | Introduce a dedicated platform service layer for search, notifications, and action dispatch before adding more page features | These behaviors were starting to duplicate across AI, event, subject, and device centers; centralizing them now reduces future drift and makes later backend integration easier | shared platform layer, AI center, event center, subject center, device center | Replace heuristic routing and static notifications with backend-backed contracts in later P6 slices |
| 2026-03-22 | D-013 | Implement the first backend-driven global search contract as a frontend aggregation layer over existing subject, device, and event list APIs | The backend does not yet expose one dedicated platform search endpoint, but operators still need real search results now; aggregating existing APIs gives us a truthful contract without blocking the upgrade | shared platform layer, subject center, device center, event center, AI center | Replace the aggregated contract with a dedicated backend search endpoint and add a result picker when the backend matures |
| 2026-03-22 | D-014 | Introduce a backend-driven notification adapter over existing exception, AI abnormal, subject, and device APIs before waiting for a dedicated notifications endpoint | Operators need real notification content now, and reusing current entity APIs keeps the shell stable while replacing static examples with live backend-backed reminders | shared notification layer, AI center, event center, backend contract sequencing | Replace the adapter with a dedicated notifications endpoint when the backend exposes one and extend it to more centers |
| 2026-03-22 | D-015 | Require operator selection when backend-driven search returns multiple plausible results instead of auto-opening the top match | Backend-backed search improves truthfulness, but auto-jumping on ambiguous matches can still break operator trust and workflow continuity | global search UX, shared platform services, subject center, device center, event center, AI center | Add a richer result picker later if the simple selector flow proves too limiting |
| 2026-03-22 | D-016 | Document one canonical `/platform/search` backend endpoint to replace the current frontend aggregation layer | The platform now has a working search contract, so the next risk is divergence between frontend orchestration and future backend behavior; a concrete endpoint spec reduces that ambiguity early | backend coordination, shared platform layer, search roadmap | Switch the frontend to the dedicated endpoint behind a safe compatibility layer when the backend implementation is ready |
| 2026-03-22 | D-017 | Extract route-query hydration and selected-row restoration into a shared composable instead of keeping three page-local copies | The subject, device, and event centers had converged on the same synchronization flow; centralizing it reduces drift while keeping field mapping and match rules page-local | shared composables, subject center, device center, event center | Reuse the composable for future list-detail centers only when they truly share the same route-sync pattern |
| 2026-03-22 | D-018 | Make `/platform/search` the preferred frontend search path and keep the aggregated entity-list search as a compatibility fallback | This lets the frontend align with the canonical backend contract now without breaking environments where the dedicated endpoint is not yet fully available or verifiable | shared search layer, backend coordination, operator search UX | Remove the fallback after authenticated production validation confirms the dedicated endpoint is stable |
